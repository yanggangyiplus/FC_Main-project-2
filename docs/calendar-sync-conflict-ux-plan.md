# 캘린더 동기화 충돌 해결 UX 기획서

## 1. 현재 상태 분석

### 1.1 현재 동기화 구조
```
┌─────────────────┐          ┌─────────────────┐
│   Always Plan   │  ◄────►  │ Google Calendar │
│   (웹앱 Todo)   │   동기화  │                 │
└─────────────────┘          └─────────────────┘
        │                            │
        ▼                            ▼
   Todo 모델                    Google Event
   - google_calendar_event_id   - extendedProperties.private.sourceId
   - bulk_synced               - (Always Plan Todo ID 저장)
```

### 1.2 현재 동기화 방식
| 기능 | 설명 |
|------|------|
| **Export (내보내기)** | 웹앱 일정 → Google Calendar |
| **Import (가져오기)** | Google Calendar → 웹앱에 표시 (읽기 전용) |
| **일괄 동기화** | 기존 모든 일정을 한번에 동기화 |
| **매칭 로직** | "제목_날짜_시간" 키로 기존 이벤트 매칭 |

## 2. 충돌 시나리오 분류

### 2.1 데이터 충돌 (Data Conflicts)

#### 충돌 유형 A: 양방향 수정 충돌
```
시나리오: 같은 일정을 양쪽에서 동시에 수정

시간순서:
1. [10:00] 웹앱에서 "팀 미팅" 일정 생성 → Google Calendar 동기화
2. [10:05] 사용자가 Google Calendar에서 시간을 14:00→15:00로 변경
3. [10:06] 사용자가 웹앱에서 제목을 "팀 미팅"→"분기 미팅"으로 변경
4. [10:07] 동기화 시도 → 충돌 발생!

현재 문제:
- 어느 쪽 변경사항을 우선할지 결정 로직 없음
- 사용자에게 충돌 알림 없음
```

#### 충돌 유형 B: 삭제 vs 수정 충돌
```
시나리오: 한쪽에서 삭제, 다른 쪽에서 수정

케이스 1 - 웹앱 삭제 + Google 수정:
1. 웹앱에서 일정 삭제
2. Google Calendar에서 해당 일정 수정 (삭제 전)
3. 동기화 시 → 삭제된 일정에 대한 수정 요청 실패

케이스 2 - Google 삭제 + 웹앱 수정:
1. Google Calendar에서 일정 삭제
2. 웹앱에서 해당 일정 수정
3. 동기화 시 → 존재하지 않는 이벤트 업데이트 시도
```

#### 충돌 유형 C: 중복 생성
```
시나리오: 동일한 일정이 양쪽에 각각 생성

1. 오프라인 상태에서 웹앱에 일정 추가
2. 동일한 일정을 Google Calendar에 직접 추가
3. 온라인 복귀 후 동기화 → 중복 일정 발생

현재 대응:
- "제목_날짜_시간" 매칭으로 일부 중복 방지
- 하지만 시간이 조금이라도 다르면 중복 생성됨
```

### 2.2 기술적 충돌 (Technical Conflicts)

#### 충돌 유형 D: 네트워크 오류로 인한 부분 동기화
```
시나리오: 동기화 중 네트워크 끊김

1. 10개 일정 일괄 동기화 시작
2. 5개 성공 후 네트워크 오류
3. 웹앱: 5개만 google_calendar_event_id 저장됨
4. 재시도 시 → 이미 동기화된 5개 중복 생성 위험
```

#### 충돌 유형 E: 토큰 만료
```
시나리오: Google OAuth 토큰 만료

1. 사용자가 장시간 미접속
2. 토큰 만료됨
3. 일정 수정 시 동기화 실패
4. 사용자 인지 못함 → 데이터 불일치
```

### 2.3 논리적 충돌 (Logical Conflicts)

#### 충돌 유형 F: 반복 일정 불일치
```
현재 상태: 반복 규칙은 Google Calendar로 전달하지 않음

문제:
1. 웹앱에서 "매주 월요일 팀 미팅" 반복 일정 생성
2. Google Calendar에는 단일 이벤트로만 저장
3. 웹앱에서 반복 규칙 수정 → Google의 개별 이벤트들과 불일치
```

#### 충돌 유형 G: 시간대 불일치
```
시나리오: 다른 시간대에서 일정 생성/수정

1. 서울(UTC+9)에서 일정 생성: 14:00
2. 미국 출장 중 Google Calendar에서 확인: 01:00 (EST)
3. 미국에서 시간 수정
4. 귀국 후 시간대 혼란 발생
```

## 3. 충돌 해결 전략

### 3.1 충돌 감지 메커니즘

#### 제안: 버전 관리 시스템 도입
```
Todo 모델 확장:
- last_modified_at: DateTime       # 마지막 수정 시간
- sync_version: Integer            # 동기화 버전
- last_synced_at: DateTime         # 마지막 동기화 시간
- sync_status: Enum                # pending, synced, conflict, error
- conflict_data: JSON              # 충돌 시 양측 데이터 저장
```

#### 충돌 감지 로직
```python
def detect_conflict(local_todo, google_event):
    local_modified = local_todo.last_modified_at
    google_modified = parse_google_updated_time(google_event['updated'])
    last_sync = local_todo.last_synced_at

    # 양쪽 모두 마지막 동기화 이후 수정됨 → 충돌
    if local_modified > last_sync and google_modified > last_sync:
        return ConflictType.BOTH_MODIFIED

    # 로컬만 수정됨 → 로컬 우선
    if local_modified > last_sync:
        return ConflictType.LOCAL_WINS

    # 구글만 수정됨 → 구글 우선
    if google_modified > last_sync:
        return ConflictType.GOOGLE_WINS

    return ConflictType.NO_CONFLICT
```

### 3.2 충돌 해결 정책 (Conflict Resolution Policy)

#### 옵션 1: 자동 해결 (권장 기본값)
```
정책: "마지막 수정 우선 (Last Write Wins)"

장점:
- 사용자 개입 최소화
- 구현 단순

단점:
- 먼저 수정한 내용 유실 가능

적용 케이스:
- 사소한 변경 (제목, 설명 등)
- 시간 변경 (동일 날짜 내)
```

#### 옵션 2: 사용자 선택 (충돌 시)
```
정책: 충돌 발생 시 사용자에게 선택권 제공

UI 플로우:
1. 충돌 감지 → sync_status = 'conflict'
2. 캘린더 화면에 충돌 뱃지 표시
3. 탭 시 충돌 해결 모달 표시
4. 사용자가 버전 선택 또는 병합
```

#### 옵션 3: 소스 우선순위 (Source Priority)
```
설정 옵션:
- "Always Plan 우선": 웹앱 변경사항 항상 우선
- "Google Calendar 우선": Google 변경사항 항상 우선
- "수동 확인": 모든 충돌 직접 확인
```

-- sync_status: pending, syncing, synced, conflict, error
ALTER TABLE todos ADD COLUMN conflict_data TEXT;

## 7. 사용자 시나리오별 플로우

### 시나리오 1: 일반적인 충돌 해결
```
1. 사용자가 캘린더 화면 진입
2. 충돌 있는 일정에 ⚠️ 뱃지 표시
3. 일정 탭 → 충돌 해결 모달 표시
4. "Always Plan 버전 사용" 선택
5. 선택한 버전으로 양쪽 업데이트
6. ✓ 동기화됨 표시
```

### 시나리오 2: 삭제 충돌
```
1. Google에서 삭제된 일정을 웹앱에서 수정 시도
2. "이 일정은 Google Calendar에서 삭제되었습니다" 알림
3. 옵션:
   - "웹앱에서도 삭제"
   - "Google Calendar에 다시 추가"
4. 선택에 따라 처리
```

### 시나리오 3: 오프라인 → 온라인
```
1. 오프라인 상태에서 여러 일정 수정
2. 온라인 복귀 시 자동 동기화 시작
3. 충돌 발생 시 알림: "3개의 충돌이 발생했습니다"

```
